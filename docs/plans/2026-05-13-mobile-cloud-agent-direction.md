## 背景

Lumina Note 当前是一个 local-first 的桌面知识工具。现有的移动端方向不应继续被理解为”必须做完整原生客户端”。

在记录基准提交下，仓库里的 `mobile/` 目录包含早期的 iOS SwiftUI 和 Android Kotlin/Jetpack Compose companion app 尝试。继续维护两套原生客户端，会把产品拖向”移动版完整 Lumina Note”，而这不是当前最低成本、最高价值的路径。该目录下的代码可视为可废弃的历史实现，不再作为后续移动端主线。

当前真正需要回答的问题是：

> 怎样以最低成本，让 Lumina 的用户在桌面之外也能访问、阅读和使用自己的知识库？

## 核心判断

移动端不应该是强编辑器，而应该是重阅读、轻交互的知识入口。

手机端最有价值的场景不是长文写作、复杂 Markdown 编辑、block 操作或 PDF 批注，而是：

- 随时阅读和浏览自己的 Markdown 笔记。
- 从知识库中检索和召回信息。
- 与 AI Agent 对话，围绕知识库提问、总结、整理。
- 快速采集新的想法、链接和片段。
- 把需要深度处理的任务留给桌面端。

第一版移动端可以不提供用户手动编辑能力。但 AI Agent 可以在云端对文档进行整理、生成和改写——这是”轻编辑”的含义：用户不直接在移动端编辑，Agent 代替用户在云端完成知识整理工作。

## 产品形态：Lumina Cloud

Lumina Cloud 是 Lumina Note 的云端付费服务。用户注册并订阅后，获得两层核心价值：

**第一层：云同步与存储。** 用户可以将本地 Markdown 知识库同步到 Lumina Cloud，获得云端存储空间。这本身就是付费价值——跨设备访问自己的笔记、数据安全备份、随时随地阅读。

**第二层：AI Agent 能力。** 在云端知识库基础上，用户可以使用 AI Agent。Agent 能对话、召回、总结、整理和生成 Markdown 内容。AI 能力按月提供使用额度。

两层价值并列，不是主从关系。存储和同步不只是”基础设施”——对很多用户来说，能在手机上阅读自己的笔记、跨设备访问知识库，这件事本身就值得付费。AI Agent 是在这个基础上叠加的更高价值。

第一阶段只面向 Markdown 知识库，不扩展到通用文件理解、Office 文档、PDF、OCR、图片识别或”上传任何文件”的场景。先把 Markdown 的同步、阅读、召回和 Agent 工作流做清楚，再考虑是否扩展其他格式。

这个产品不应该被定位成通用网盘。如果说成”网盘”或”云盘”，用户会拿它和 iCloud、Google Drive、Dropbox 比较容量、速度、文件分享和价格。更准确的定位是：

> Markdown 知识云：同步笔记、随时阅读、AI 帮你理解和整理。

## 注册与多渠道适配

用户需要注册 Lumina 账号才能使用云服务。目标用户群体以国内用户为主，注册渠道需要优先适配国内主流方式：

优先级从高到低：

1. 手机号码（短信验证码登录，国内用户最通用的方式）。
2. 微信登录。
3. QQ 登录。
4. 未来可考虑 Apple、Google 等海外渠道。

注册入口需要覆盖：

- 桌面端：应用内注册和登录。
- 移动 Web / PWA：网页端注册和登录。
- 未来如有小程序或原生 App：对应平台的注册流程。

代码侧需要先把账号体系、认证流程和多渠道登录的架构搭好。第三方渠道的真实接入可能需要一段时间（申请开发者资质、审核等），但代码结构不应该因此耦合——认证层应该设计为可插拔的 provider 模式，新增一个登录渠道只需要实现一个 provider，不需要改动核心认证逻辑。

账号体系是跨端体验的基础，需要在第一版就设计好。

## 多端同步

当云端文档发生变更（无论是用户在另一台设备上操作，还是 AI Agent 在云端整理了内容），其他端需要同步这些变更。

这是一个已经被行业充分解决的问题。Obsidian Sync、iCloud Drive、Dropbox 等产品已经建立了用户对”多端同步”的成熟心智预期：改了就同步，有冲突就提示，离线了回来再合并。

Lumina 不需要发明新的同步模式，只需要做到行业成熟水平：

- 云端变更能及时推送到其他已连接的端。
- 桌面端本地变更能同步到云端。
- 冲突处理有明确策略（时间戳优先、或保留双方版本供用户选择）。
- 离线场景下能暂存变更，上线后自动合并。

这是一个工程量不小但方案成熟的子系统，不存在需要探索的未知问题。

## 移动端 MVP

第一版移动端以阅读和 AI 交互为中心。

主流程：

- 浏览和阅读云端同步的 Markdown 笔记。
- 向 AI Agent 提问，Agent 从知识库中召回相关内容并回答。
- 回答中展示引用来源，用户可以点开查看原文。
- 快速采集想法、链接、粘贴文本到 inbox。
- 让 Agent 对云端文档进行整理、总结或生成新内容。

次级流程：

- 查看最近笔记。
- 查看 inbox。
- 查看账号、套餐和额度。

第一版明确不做：

- 用户手动编辑 Markdown（阅读优先，编辑留给桌面端）。
- 完整 CodeMirror 编辑体验。
- 知识图谱。
- PDF 批注。
- 插件系统。
- 完整离线同步。
- 原生文件系统管理。
- PDF、Word、PPT、表格、图片 OCR 或任意文件解析。

## 技术方向

最低成本路径应该是 Web-first：

- 先做移动 Web / PWA。
- 尽量复用现有 React、TypeScript、Zustand 和服务层代码。
- Markdown 索引、语义检索和 Agent 执行放在云端。
- 移动端只作为阅读和交互入口，不承担 local-first 的复杂性。

如果移动 Web 体验被验证有价值，再考虑用 Capacitor 把同一套移动 Web 包成 iOS / Android app。这条路线更接近 Obsidian 的跨端模式，而不是继续维护 SwiftUI + Kotlin 两套原生客户端。

当前 `mobile/` 原生方向不作为后续主线。

## 小程序定位

微信小程序不应作为主客户端。

它可以作为未来的轻入口或采集渠道：

- 分享文本或链接到 Lumina。
- 快速采集内容到 inbox。
- 做简单问答。
- 绑定账号或扫码登录。

但它不应该承载核心知识空间和完整体验。平台限制会让它看起来便宜、实际变贵。

## 商业模型

订阅对应两类真实成本：

**存储与同步：** 用户注册后获得一定的免费存储额度。付费订阅解锁更大的存储空间和完整的同步能力。

**AI Agent：** 付费用户每月获得 AI 使用额度。不同复杂度的任务消耗不同额度——简单问答和长上下文多轮召回、生成结构化笔记的任务不等价。

具体的套餐档位、存储额度和 AI 使用额度尚未确定。代码侧需要预留完整的套餐与额度管理能力：

- 套餐体系支持多档位，每档可配置存储上限、AI 月度额度等参数。
- 额度检查和扣减逻辑需要从业务逻辑中抽离，作为独立的中间件或服务层存在。
- 套餐参数应该是可配置的（数据库或配置文件），不应硬编码在业务逻辑中。
- 需要支持免费试用额度，让未付费用户也能体验云同步和 Agent 能力。

大致的方向是分为 Free / Pro / Power 三档，但具体数字和定价后续确定。

## 与桌面端的关系

桌面端仍然是深度工作环境：

- 长文写作和编辑。
- local-first vault 管理。
- 复杂 Agent 任务。
- PDF 工作流。
- 知识图谱和结构整理。
- 插件生态。

移动端是随时访问环境：

- 阅读。
- 检索和召回。
- 与 Agent 对话。
- 快速采集。
- 深度工作交回桌面。

云端是连接层：

- 存储和同步用户的 Markdown 知识库。
- 运行 AI Agent。
- 管理索引和检索。
- 提供跨设备的统一数据层。

## 执行约束与反模式

这份文档会直接作为 Claude Code agent 的执行目标。以下约束必须严格遵守。

### 任务必须可闭环验证

每个任务的核心链路必须是 Claude 自己可以端到端验证的。如果一个任务写完代码后无法通过运行测试、启动服务、发送请求等方式自行确认"这条链路是通的"，那这个任务的拆分有问题。

- 写完一个 API 端点，必须能用测试或 curl 验证它返回正确结果。
- 写完一个认证流程，必须能用测试验证 token 签发、校验、拒绝的完整路径。
- 写完一个同步逻辑，必须能用测试验证变更传播和冲突处理。
- 如果某个环节依赖外部服务（如短信发送、微信 OAuth），用可切换的 mock provider 做本地验证，但 mock 本身必须严格模拟真实行为的输入输出契约，不能跳过核心逻辑。

### 禁止的代码模式

**禁止空实现。** 不允许写一个函数签名然后 `throw new Error('not implemented')` 或返回空值。如果一个功能还没准备好做，就不要创建它的接口。已经存在的接口必须有完整实现。

**禁止玩具实现。** 不允许用 in-memory 数组假装数据库、用 `setTimeout` 假装异步任务队列、用全局变量假装状态管理。每一层都必须用生产级的技术选型。如果当前阶段只需要 SQLite，那就用真实的 SQLite，不要用 JSON 文件假装持久化。

**禁止幼稚实现。** 不允许忽略并发、忽略错误边界、忽略资源清理。不允许写出"只在 happy path 下能工作"的代码。认证不能只检查 token 存在而不验证签名。额度扣减不能不考虑并发竞争。文件同步不能不处理冲突。

**禁止绕过验证。** 不允许为了让测试通过而降低测试标准。不允许 `// @ts-ignore` 或 `as any` 绕过类型检查。不允许 catch 错误后静默吞掉。

### 允许重构既有基础设施

如果现有代码中的基础设施（同步、云服务、认证、存储等）实现质量差或架构不合理，允许且鼓励重构，甚至完全重写。不要在一个糟糕的地基上继续堆砌新功能。

判断标准：如果在既有实现上做新功能，需要不断绕开旧代码的设计缺陷或写大量适配层，那就应该先重构旧代码，再做新功能。沉没成本不是保留烂代码的理由。

### 不确定最佳实践时怎么办

如果对某个领域的最佳实践没有把握——例如 JWT 刷新策略、文件同步冲突解决算法、额度扣减的并发控制——**不要猜测，去找参考实现**。

具体做法：

- 在 GitHub 上找到该领域公认优秀的开源项目。
- clone 下来，阅读它的核心实现。
- 理解它为什么这样设计，然后在 Lumina 的上下文里做适配。
- 在代码注释中标注参考来源。

宁可花时间研究成熟方案，也不要从零发明一个看似合理但未经验证的实现。

### 产品与工程通识原则

以下是容易被 AI 忽视但在生产系统中至关重要的通识原则。

**宁可明确报错，也不要静默失败。** 额度用完了，告诉用户"本月额度已用完"，不要静默降级到返回空结果。同步失败了，告诉用户"同步失败，稍后重试"，不要假装成功。API 调用超时了，返回明确的错误码和提示，不要返回一个空的 200。用户宁愿看到一个清晰的错误提示，也不愿意面对一个"好像工作了但结果不对"的系统。

**区分用户错误和系统错误。** 用户输入了非法手机号，这是 400，提示应该告诉用户怎么修正。服务器内部崩溃了，这是 500，提示应该是"系统繁忙，请稍后重试"，同时内部记录完整的错误信息用于排查。不要把堆栈信息暴露给用户，也不要把所有错误都笼统地说"出错了"。

**幂等性。** 所有会改变状态的操作（扣减额度、创建笔记、触发同步）必须考虑重复执行的情况。网络抖动导致客户端重试是常态，不是异常。一次扣减请求重试三次不应该扣三次额度。

**速率限制。** 所有对外暴露的 API 必须有速率限制。没有速率限制的 API 就是一个等待被滥用的漏洞。包括注册接口（防刷号）、短信验证码接口（防轰炸）、AI 调用接口（防超额）。

**敏感数据处理。** 用户密码必须哈希存储，永远不存明文。Token 必须有过期时间。日志中不能出现用户密码、完整 token、笔记正文等敏感信息。API 响应中不返回多余的用户信息。

**状态变更必须有审计线索。** 额度扣减、套餐变更、文档删除等关键操作需要记录操作日志（谁、什么时间、做了什么、结果是什么）。这不是为了合规报表，而是出了问题时能回溯。

**优雅降级有边界。** 降级是指在部分能力不可用时仍然提供核心服务，比如 AI 服务宕机时同步和阅读仍然可用。但降级不是静默吞错——降级时必须让用户知道哪些功能暂时不可用，而不是默默返回残缺的结果。

### 技术选型原则

- 优先选择在国内生态中成熟、文档充分、社区活跃的方案。
- 数据库、消息队列、缓存等基础设施选型要考虑部署在国内云服务商的场景。
- 认证相关的第三方 SDK 优先考虑国内渠道的官方 SDK。

### 实现节奏

按以下顺序推进，每个阶段的产出都应该是可运行、可验证的，后一个阶段建立在前一个阶段的基础上。

**阶段一：最小端到端链路。** 用户注册 → 上传一篇 Markdown 笔记 → 在移动 Web 上阅读它。这条链路跑通意味着账号体系、云存储、API、移动 Web 渲染都能工作。这是后续所有功能的地基。

**阶段二：认证与多渠道登录。** 在阶段一的账号体系基础上，完善可插拔的 provider 认证架构。实现手机号验证码登录（用 mock provider 本地验证）、JWT 签发与刷新、会话管理。

**阶段三：云同步。** 桌面端本地 vault 与云端的双向同步。变更检测、增量同步、冲突处理。这是让"多端访问同一份知识库"成为现实的核心子系统。

**阶段四：套餐与额度体系。** 可配置的多档位套餐、存储额度检查、AI 使用额度扣减与月度重置。额度中间件对所有需要计量的 API 生效。

**阶段五：AI Agent 集成。** 云端 Agent 能力——基于用户 Markdown 知识库的对话、召回、总结、整理和内容生成。Agent 对文档的修改通过阶段三的同步机制传播到其他端。

**阶段六：移动 Web 完善。** 在阶段一的移动 Web 基础上，补全笔记浏览、检索、Agent 对话、inbox 采集等 MVP 流程。优化移动端阅读体验。

各阶段之间不是严格的瀑布关系——如果某个阶段的依赖已经就绪，可以并行推进。但核心原则是：先让地基稳固，再往上建。

## 当前决策

> 不做完整移动编辑器。移动端重阅读、轻交互，第一版不提供用户手动编辑。AI Agent 可在云端对文档做整理和生成。云同步与 AI 能力是两层并列的付费价值。以移动 Web / PWA 验证交互模型，之后再考虑 Capacitor 打包。多端同步采用行业成熟方案。注册以手机号、微信、QQ 为优先渠道，面向国内用户。套餐档位和额度待定，代码侧预留完整的配置化能力。小程序只作为未来轻入口，不作为主客户端。`mobile/` 原生代码可废弃，不作为后续主线。每个实现任务必须可闭环验证，禁止空实现、玩具实现和幼稚实现。
